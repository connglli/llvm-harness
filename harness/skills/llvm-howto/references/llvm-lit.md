# llvm-lit — LLVM Regression Test Runner

`llvm-lit` is LLVM's portable test execution tool. It discovers and runs
regression tests, summarizes results, and reports failures. Tests use the
`FileCheck` pattern-matching system to verify tool output.

## Basic Usage

```bash
# Run all tests in a directory
llvm-lit -sv llvm/test/Transforms/InstCombine

# Run a single test file
llvm-lit -sv llvm/test/Transforms/InstCombine/add.ll

# Run tests matching a pattern
llvm-lit -sv --filter='add' llvm/test/Transforms/InstCombine

# Stop after first failure (fast debugging)
llvm-lit -sv --max-failures=1 llvm/test/Transforms/InstCombine
```

## Flags Reference

### Output Control

| Flag | Description |
|------|-------------|
| `-q`, `--quiet` | Suppress all output except failures |
| `-s`, `--succinct` | Show progress bar, only print failures |
| `-v`, `--verbose` | Show test names and commands as they run |
| `-vv` | Deprecated alias for `-v` |
| `-a`, `--show-all` | Enable `-v` for all tests, not just failures |
| `--no-progress-bar` | Disable the curses progress bar |
| `--show-unsupported` | Show tests that were skipped |
| `--show-xfail` | Show tests that failed as expected |

### Execution Control

| Flag | Description |
|------|-------------|
| `-j <N>` | Run N tests in parallel (default: CPU count) |
| `--max-failures=<N>` | Stop after N failures |
| `--filter=<regex>` | Only run tests whose name matches the regex |
| `--order=lexical` | Alphabetical order (reproducible) |
| `--order=random` | Random order (find order-dependent bugs) |
| `--time-tests` | Report execution time per test |
| `--no-execute` | Don't run tests, just discover them |
| `--xunit-xml-output=<file>` | Write JUnit XML report |
| `-D NAME=VALUE` | Pass a user parameter to the test suite |

### Debugging

| Flag | Description |
|------|-------------|
| `-v` | Prints each `RUN:` command before execution — the last printed command is the failing one |
| `--vg` | Run tests under Valgrind (memcheck) |
| `--vg-leak` | Enable Valgrind memory leak checks |

## Understanding Test Files

A typical LLVM regression test:

```llvm
; NOTE: Assertions have been autogenerated by utils/update_test_checks.py
; RUN: opt -passes=instcombine -S < %s | FileCheck %s

define i32 @test(i32 %x) {
; CHECK-LABEL: @test(
; CHECK-NEXT:    ret i32 0
  %sub = sub i32 %x, %x
  ret i32 %sub
}
```

### RUN Lines

- `RUN:` lines define shell commands to execute
- `%s` is replaced with the test file path
- `%t` is a temporary file path
- `%S` is the directory containing the test file
- Multiple `RUN:` lines are executed sequentially; all must pass

### FileCheck Directives

| Directive | Meaning |
|-----------|---------|
| `CHECK:` | Match this string somewhere in the output |
| `CHECK-NEXT:` | Match on the very next line after the previous match |
| `CHECK-SAME:` | Match on the same line as the previous match |
| `CHECK-NOT:` | Assert this string does NOT appear before the next match |
| `CHECK-LABEL:` | Anchor point — resets the match position |
| `CHECK-DAG:` | Match in any order (unordered group) |
| `CHECK-EMPTY:` | Match an empty line |
| `CHECK-COUNT-<N>:` | Match exactly N consecutive occurrences |

### Check Prefixes

Use `--check-prefix` to have multiple CHECK groups in one file:

```
; RUN: opt -passes=instcombine -S < %s | FileCheck %s --check-prefix=IC
; RUN: opt -passes=gvn -S < %s | FileCheck %s --check-prefix=GVN
; IC: ...
; GVN: ...
```

### Special Directives

| Directive | Meaning |
|-----------|---------|
| `REQUIRES:` | Test only runs if the feature is available |
| `UNSUPPORTED:` | Test skipped if the feature is present |
| `XFAIL:` | Test is expected to fail |

```
; REQUIRES: asserts
; XFAIL: target=arm{{.*}}
```

## Common Test Patterns

### Standard pass test

```
; RUN: opt -passes=instcombine -S < %s | FileCheck %s
```

### Expect a crash (negative test)

```
; RUN: not opt -passes=instcombine -S < %s 2>&1 | FileCheck %s
; CHECK: LLVM ERROR
```

### Expect a specific assertion failure

```
; RUN: not --crash opt -passes=instcombine -S < %s 2>&1 | FileCheck %s
; CHECK: Assertion {{.*}} failed
```

### Multiple RUN lines with different configs

```
; RUN: opt -passes=instcombine -S < %s | FileCheck %s --check-prefix=CHECK1
; RUN: opt -passes=gvn -S < %s | FileCheck %s --check-prefix=CHECK2
```

### Regex matching in FileCheck

```
; CHECK: %[[VAR:.*]] = add i32
; CHECK-NEXT: ret i32 %[[VAR]]
```

## Auto-Generating CHECK Lines

LLVM provides scripts to automatically generate CHECK lines:

```bash
# For opt tests
python llvm/utils/update_test_checks.py \
  --opt-binary=<build>/bin/opt \
  llvm/test/Transforms/InstCombine/add.ll

# For llc tests
python llvm/utils/update_llc_test_checks.py \
  --llc-binary=<build>/bin/llc \
  llvm/test/CodeGen/X86/add.ll
```

These rewrite the file in-place with CHECK lines matching the current
tool output. Always review the generated lines before committing.

## Test Result Status

| Status | Meaning |
|--------|---------|
| PASS | Test passed |
| FAIL | Test failed |
| XFAIL | Expected failure (test is known broken) |
| XPASS | Unexpected pass (was XFAIL but now passes) |
| UNSUPPORTED | Skipped (missing required feature) |
| TIMEOUT | Test exceeded time limit |
| UNRESOLVED | Test could not be run (setup error) |

## Test Directories

| Directory | What it tests |
|-----------|--------------|
| `llvm/test/Transforms/` | Middle-end optimization passes |
| `llvm/test/Analysis/` | Analysis passes (not transforms) |
| `llvm/test/CodeGen/` | Backend code generation (per-target subdirs) |
| `llvm/test/Assembler/` | IR assembler/parser |
| `llvm/test/Verifier/` | IR verifier |
| `llvm/test/Bitcode/` | Bitcode reader/writer |
| `llvm/test/Linker/` | IR linker |
| `llvm/test/DebugInfo/` | Debug information |
